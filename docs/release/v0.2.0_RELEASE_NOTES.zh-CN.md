# Retinue v0.2.0 发布说明

Retinue v0.2.0 是 OpenCode-native 加固周期后的第一个发布候选。默认产品路径仍然是 OpenCode；Retinue 不再给普通 OpenCode 子代理叠加自造的 read-only access-mode 层，而是尽量复用 OpenCode 自身的 agent/profile/permission 语义，并把权限请求和 stalled 诊断显式上浮给调用方。

英文发布说明见 [v0.2.0_RELEASE_NOTES.md](v0.2.0_RELEASE_NOTES.md)。

## v0.1.0 之后的主要变化

- OpenCode 调用优先使用官方 `@opencode-ai/sdk`；旧 HTTP client 只保留为兼容回退。
- 新增后端中立的权限工具：
  - `retinue_list_permissions`
  - `retinue_reply_permission`
- OpenCode `external_directory` 权限请求会以结构化 approval guidance 上浮，而不是只表现为一个不可解释的 stalled job。
- Claude Code 支持 Claude Agent SDK 路径，包括 SDK permission hook。Retinue 默认不选择 Claude 模型；Claude Code 的 profile、endpoint 和模型路由仍由 Claude Code 自己管理，除非部署显式设置 `RETINUE_CLAUDE_MODEL`。
- OpenCode provider 和 tool-call 失败都会进入有界 stalled 诊断。可恢复失败可以启动一次 fresh task-level attempt；原始失败链仍然是非证据。
- 日志审计支持 live reconcile：可以用当前 job/backend 状态压掉陈旧 stalled 记录，忽略 completed jobs，去重 recovery chain，输出简洁 issue candidate，并把权限等待从后端失败里分离成 attention item。
- 安装缓存里的 `retinue.config.json` 被定义为包默认值，不是持久用户配置。持久覆盖应写环境变量，例如 `RETINUE_MAX_CONCURRENT_AGENTS`、`RETINUE_OPENCODE_AGENT`、`RETINUE_OPENCODE_ROOT_BINDING_MODE`。
- Kilo 和 backend-candidate probe 已加入评估路径，但 v0.2.0 默认产品后端仍不是 Kilo。

## 产品边界

Retinue 仍然是本地子代理执行面，不是 provider router，也不拥有模型选择。

- 默认后端：OpenCode。
- 默认 OpenCode agent：`explore`。
- 包默认 active child 槽位：`3`；默认共享机器级预算是 `max(5, RETINUE_MAX_CONCURRENT_AGENTS)`。部署可以用 `RETINUE_MAX_CONCURRENT_AGENTS` 覆盖单 MCP 会话上限，用 `RETINUE_GLOBAL_AGENT_BUDGET` 覆盖共享机器级预算。
- Overflow 默认改为 `RETINUE_OVERFLOW_STRATEGY=queue`：session/global active 槽位满时返回 `queued` job handle，并在空槽出现后 promote。`RETINUE_OVERFLOW_STRATEGY=evict` 保留旧的同 session 最旧 running 子代理驱逐行为。`RETINUE_MAX_QUEUED_AGENTS` 默认 `20`；队列满时返回 `resource_exhausted` 和 `reason: "queue_full"`。
- 默认 OpenCode root binding mode 仍是 `per-spawn`。`shared-root` 可通过 `RETINUE_OPENCODE_ROOT_BINDING_MODE=shared_root` 作为部署实验开启。
- backend、profile、model、OpenCode server、access mode、bash policy 不是普通产品工具参数。

## 可靠性与运维

- 权限请求现在是明确的 MCP workflow event，不再只是不可解释的 stalled；权限等待响应会压缩 raw stderr，避免淹没需要主代理处理的 supervisor action。
- provider 和 tool-call 失败会被有界地归类为 stalled 诊断，而不是被当作可信结果。
- 日志审计能力面向维护者保留，但诊断工具不进入默认产品工具面。
- 发布验证由仓库 gate 和维护者 runbook 记录，不写入公开 release notes。

## 升级注意

- 如果你曾直接编辑包内 `retinue.config.json`，请把持久配置迁移到环境变量。插件刷新可能覆盖包内配置文件。
- 如果需要更多并发子代理，在 Codex host 或 MCP server 环境中设置 `RETINUE_MAX_CONCURRENT_AGENTS`。
- 如果开启 shared OpenCode root，请用 `externalRunnerMode`、`externalRootSessionId` 和 child `externalSessionId` 区分日志，避免和 per-spawn 结果串台。
