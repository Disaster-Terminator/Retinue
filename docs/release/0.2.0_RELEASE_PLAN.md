# Retinue 0.2.0 Release Readiness

This document records the current release-prep boundary for v0.2.0. It is not the publish action; tag creation and package publishing should happen only after the release gate passes on the final commit.

Chinese release notes are available at [v0.2.0_RELEASE_NOTES.zh-CN.md](v0.2.0_RELEASE_NOTES.zh-CN.md).

## Release Boundary

Retinue 0.2.0 is the OpenCode-native hardening release.

The release boundary is:

- Plugin marketplace installation remains the default user path.
- OpenCode remains the default backend and `explore` remains the default child agent.
- Retinue uses the official OpenCode SDK first, with the legacy HTTP client kept as a compatibility fallback.
- Retinue follows OpenCode profile, model, provider, and agent semantics instead of exposing product-level `access_mode` or `bash_policy` arguments.
- Retinue exposes backend-neutral permission list/reply tools so headless callers can handle backend permission requests explicitly.
- Claude Code has an SDK-first backend path with permission-hook support, but it is not the default backend.
- Kilo/backend-candidate support remains an evaluation path, not the default product path.

## Released Tool Surface

Retinue 0.2.0 exposes these product MCP tools:

- `retinue_spawn_agent`
- `retinue_wait_agent`
- `retinue_close_agent`
- `retinue_list_agents`
- `retinue_list_permissions`
- `retinue_reply_permission`

These tools do not accept backend, profile, model, OpenCode server, `access_mode`, `bash_policy`, or permission-mode choices as normal per-call arguments. Those choices belong to deployment configuration or the selected backend runtime. A spawn may pass `agent` to choose the OpenCode agent for that child.

Diagnostic tools such as log audit remain opt-in and should not be exposed as the default product surface for ordinary users.

## Closed Before Tag

- v0.1.0 post-release OpenCode stalls now produce bounded diagnostic `stalled` results instead of hanging indefinitely.
- Recoverable OpenCode backend failures can start one fresh task-level attempt; original stalled attempts remain non-evidence.
- OpenCode `external_directory` permission requests are surfaced through `retinue_list_permissions` and `retinue_reply_permission`.
- Claude Agent SDK permission hooks are bridged through the same Retinue permission reply surface.
- Retinue no longer hardcodes Claude model selection; SDK profile/model routing stays with Claude Code unless explicitly overridden by deployment env.
- Packaged config is treated as fallback. Persistent overrides belong in environment variables.
- Log audit reconciles stale stalled traces against live job/backend state, ignores completed jobs, and deduplicates recovery chains.
- Deployment-level concurrency overrides are supported through environment configuration.

## Release Validation

Before tagging, the final commit should pass the repository release gate and the real-backend checks maintained by the project runbooks. Keep raw command output, probe details, local environment paths, and provider incident traces in CI/session logs rather than public release notes.

The release gate covers:

- Unit and integration coverage for the shipped MCP tool surface.
- Generated output and package-shape verification.
- Backend integration behavior.
- Permission handling.
- Diagnostics and logging behavior.
