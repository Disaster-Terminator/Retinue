# Retinue 0.1.0 Release Plan

This document tracks the final work before the first public Retinue release. Its job is to keep completed decisions archived and unfinished concerns visible.

## Release Definition

Retinue 0.1.0 is the first usable local-first Codex subagent plugin.

The release is acceptable when:

- A user can install Retinue from the Codex plugin marketplace without compiling the repository.
- The default backend is OpenCode.
- The default OpenCode agent is `plan`.
- Codex sees the Retinue skill and the `retinue_*` MCP tools.
- A WSL user can install the plugin, let Retinue manage the default OpenCode server, spawn a child agent, wait for the result, and close the child agent.
- The npm package can install the runtime for custom MCP setups.
- Deterministic tests, package verification, and real backend probes have been run and recorded.

## Completed And Archived

- `retinue_spawn_agent`, `retinue_wait_agent`, and `retinue_close_agent` exist as the product tool surface.
- Product tools do not expose backend, profile, model, agent, permission mode, or OpenCode base URL as per-call arguments.
- `retinue_wait_agent` returns the terminal result, avoiding a separate result tool in the first product surface.
- OpenCode is the default backend.
- Claude Code can be selected by deployment configuration.
- `wait` and `close` bind to the backend recorded in the job metadata, so a later environment change does not reroute an existing job.
- Retinue OpenCode fake E2E passes.
- Retinue OpenCode real E2E passes.
- Retinue Claude Code fake E2E passes.
- Retinue Claude Code best-effort real E2E passes on the local machine used for development.
- The plugin marketplace path is the primary user install path.
- npm runtime packaging is supported for custom MCP setup.

## 0.1.0 Must Finish Before Tag

- Keep source build instructions out of the quick start and link to `docs/development/SOURCE_INSTALL.md`.
- Include built `dist/` runtime in the release repository so plugin marketplace installs do not require compilation.
- Rename the npm package to `@disaster-terminator/retinue`.
- Expose Retinue bin names: `retinue`, `retinue-mcp`, and `retinued`.
- Keep compatibility bin aliases out of user-facing docs.
- Update plugin MCP defaults:
  - `RETINUE_BACKEND=opencode`
  - `RETINUE_OPENCODE_AUTO_SERVE=1`
  - `RETINUE_OPENCODE_HOST=127.0.0.1`
  - `RETINUE_OPENCODE_AGENT=plan`
- Mark the Retinue plugin as available in its marketplace metadata and document the Codex `/plugins` install step.

- Re-run deterministic gates:
  - `pnpm run typecheck`

  - `pnpm run build`

- Leave the final WSL Codex plugin smoke for the user, because it touches the user's WSL Codex environment.

## User Acceptance Smoke

The user acceptance smoke is intentionally run by the user in their WSL Codex environment.

Steps:

```bash
codex plugin marketplace add Disaster-Terminator/Retinue
```

Then open Codex, run `/plugins`, press the keyboard Right Arrow key until `[Retinue Local]` is selected, press Enter to open the `Retinue` details page, choose `Install plugin`, restart Codex, and ask:

```text
Use Retinue to spawn an OpenCode plan subagent. Ask it to reply exactly: RETINUE_OK. Wait for the result and close the child agent.
```

Acceptance:

- Retinue skill is visible.
- `retinue_spawn_agent` is callable.
- `retinue_wait_agent` returns `RETINUE_OK`.
- `retinue_close_agent` returns a terminal status.

## Open Questions And Deferred Work

These items are not allowed to disappear after 0.1.0.

### Config File

Environment variables are not a good long-term product configuration layer. 0.1.0 uses plugin MCP env defaults because that is the smallest reliable release path. A later release should add a Retinue config file.

Recommended first config shape:

```json
{
  "backend": "opencode",
  "opencode": {
    "baseUrl": "http://127.0.0.1:4096",
    "agent": "plan"
  }
}
```

Recommended paths:

- Windows: `%LOCALAPPDATA%\retinue\config.json`
- Linux/WSL: `~/.config/retinue/config.json`
- Override: `RETINUE_CONFIG=/path/to/config.json`

Precedence should be: explicit process env, config file, built-in default. Tool arguments should not select backend/profile/model/agent/permission.

### OpenCode Plan And Build

OpenCode has `plan` and `build` agents. Retinue 0.1.0 defaults to `plan` because it is the safer product default for Codex subagent delegation.

Deferred decision:

- Configure `plan` or `build` at deployment/config level.
- Do not add `agent` to `retinue_spawn_agent` unless the product deliberately changes its backend-hiding contract.

### Claude Code Profile And Permissions

Claude Code is wired and validated, but Retinue does not manage a separate Claude Code child-agent profile in 0.1.0.

Deferred work:

- Document known-good Claude Code profile requirements.
- Decide whether a low-permission profile is feasible.
- Keep real Claude Code E2E best-effort because endpoint/model stability is outside Retinue.

### Future Retinue Tools

The long-term product surface may include:

- `retinue_send_message`
- `retinue_followup_task`
- `retinue_list_agents`

These should be added only after the session continuation and listing semantics are stable across OpenCode and Claude Code.

### Internal Directory Names

Some repo-local paths still contain historical names. User-facing docs should say Retinue. Internal path cleanup can happen after 0.1.0 if it does not risk breaking plugin marketplace packaging.

### Plugin Install UX

Codex CLI 0.128 exposes marketplace add/upgrade/remove. It does not expose a separate `codex plugin install` command. The 0.1.0 path therefore uses `/plugins` in the Codex TUI for installation.

The user acceptance smoke must confirm that the plugin works after adding the marketplace, installing Retinue from `/plugins`, and restarting Codex.
