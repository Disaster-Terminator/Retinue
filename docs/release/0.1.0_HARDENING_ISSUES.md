# Retinue 0.1.0 Hardening Issues

Date: 2026-05-07

This document records issues found during real Codex plugin smoke and E2E work so they do not disappear while implementation is in progress.

## Release Boundary

0.1.0 default product path:

- Codex plugin marketplace install.
- Default backend: OpenCode.
- Default OpenCode agent: `plan`.
- Claude Code remains experimental and optional.
- New users should not need to compile Retinue or hand-edit backend configuration.

## Known Issues Under Active Fix

### 1. Plugin MCP Startup From Marketplace Cache

Status: fixed in commits `cab6ec0` and `185bd98`.

Evidence:

- Initial Codex error: `MCP startup failed: handshaking with MCP server failed: connection closed: initialize response`.
- Installed plugin cache originally had `.mcp.json` and skills but no usable runtime.
- After runtime copy, cache still failed because the MCP runtime imported dependencies unavailable from the plugin cache.

Required guardrail:

- The plugin cache must start `node ./dist/mcp.js` without relying on repository root files or root `node_modules`.

### 2. Default OpenCode Server Lifecycle

Status: implemented in source; verification in progress.

Problem:

- A one-click plugin install cannot assume the user already started `opencode serve`.
- Retinue should manage the default OpenCode server lifecycle for the plugin path.
- Explicit `SUPERVISOR_OPENCODE_BASE_URL` remains the opt-in attach path for externally managed servers.

Acceptance:

- With plugin defaults and no explicit base URL, Retinue starts OpenCode on loopback.
- The default agent remains `plan`.
- Existing explicit attach behavior still works.
- Startup failures return actionable errors.

### 3. Port Conflicts Across Windows / WSL

Status: implemented in source; verification in progress.

Problem:

- Windows and WSL can both try to use `127.0.0.1:4096`.
- A healthy OpenCode server on `4096` may belong to another OS/user environment. Auto-serve must not blindly attach to it.
- The default path needs a fallback beyond a single port.

Acceptance:

- Retinue-managed auto-serve prefers `4096`.
- If `4096` is occupied by an external service, including an external OpenCode server, Retinue tries `4097`.
- If all candidate ports are occupied by external services, Retinue reports a clear port conflict.
- If `SUPERVISOR_OPENCODE_PORT` is explicitly set, Retinue does not silently fall back to another port.

### 4. Multiple Codex Threads

Status: implemented in source; verification in progress.

Problem:

- Each Codex thread starts its own Retinue MCP process.
- In-memory tracking of a managed OpenCode server only works within one MCP process.
- Multiple Codex threads need to spawn OpenCode child agents without racing on ports or mistakenly attaching to another environment's OpenCode server.

Implemented design:

- Store Retinue-managed OpenCode server discovery under the Retinue state directory.
- Protect server startup with a cross-process lock.
- Reuse only a server recorded as Retinue-managed and still health-checking.
- Fall back to the next candidate port when the preferred port is occupied by an unowned external service.

Acceptance:

- Two independent Retinue MCP processes can both spawn OpenCode plan subagents.
- The second process reuses the managed server recorded by the first process, or starts a fallback server without port collision.
- Stale discovery is ignored and replaced.

### 5. Empty Result From Real OpenCode Plan Subagent

Status: implemented in source; verification in progress.

Evidence:

- A real Retinue delegation to OpenCode `plan` completed with empty `parsedStdout.result`.

Risk:

- E2E may report a completed job but return no useful answer to Codex.

Acceptance:

- `retinue_wait_agent` returns the assistant text from the OpenCode session for deterministic prompts.
- If OpenCode returns no assistant text, Retinue reports a structured failure or running state instead of a misleading completed empty result.

Implemented behavior:

- A completed assistant message without a visible `text` part does not transition an OpenCode job to `completed`.
- `wait` remains `running` until a visible assistant result exists or the timeout expires.

### 6. Marketplace Cache Refresh Can Leave Other Codex Sessions With Failed Plugin State

Status: observed during local smoke; product code and package guardrails updated, Codex session reload behavior remains external.

Evidence:

- Local marketplace metadata points at revision `8192b0a4000865d80c2216673c8d2fdfe67caec2`.
- The installed plugin cache and marketplace clone both contain the bundled `plugins/anchorpoint/dist/mcp.js` with OpenCode auto-serve and fallback-port support.
- A direct stdio smoke from `/home/raystorm/.codex/plugins/cache/retinue-local/retinue/0.1.0` can list tools and run an OpenCode spawn/wait/close cycle.
- Codex logs around a plugin reinstall window include `configured non-curated plugin no longer exists in discovered marketplaces during cache refresh` followed by `failed to load plugin: plugin is not installed plugin="retinue@retinue-local"`.

Risk:

- A Codex session opened during cache refresh or before plugin reload may keep reporting Retinue unavailable even after the cache is correct.

Guardrail:

- CI must fail if source changes rebuild `dist/` or `plugins/anchorpoint/dist/` without the generated artifacts being committed.
